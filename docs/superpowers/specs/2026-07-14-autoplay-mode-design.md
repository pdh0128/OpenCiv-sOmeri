# 오토모드 (Autoplay Demo Mode) 설계

**Date:** 2026-07-14
**Status:** Approved (design), pending implementation plan

## Background

소메리 마이그레이션 마스터플랜(6항목) 전부 완료. 사용자가 완성된 게임을 나중에 화면 녹화해서 영상으로 남기고 싶어함 — 사람이 계속 클릭 안 해도 게임이 알아서 진행되는 "오토모드" 필요.

## 결정된 방향

- **트리거**: URL 쿼리 파라미터 `?autoplay=true` (기존 `?test=true&scenario=X` e2e 테스트 진입 방식과 동일한 패턴, `client/src/Index.ts`에 분기 추가).
- **접속 흐름**: 기존 `TestUtils.ensureInGame()` 그대로 재사용 — localhost 접속, 로비 통과, in_game 진입까지 이미 구현되어 있음. 광역주(파벌) 선택은 하지 않음 — 기존 `LobbyState.onDestroyed()`의 미선택 플레이어 자동배정 로직에 맡김(이미 존재하는 폴백, 새로 안 만듦).
- **의사결정 수준**: 단순 규칙 기반이되 약간의 우선순위를 둠(사용자 확인):
  - **정착민**: 발견 즉시 정착(찾자마자 `settle` 액션 실행).
  - **건물 큐**: 도시가 아무것도 안 짓고 있으면, 건설 가능 목록(잠금 안 걸리고 아직 안 지어진 것) 중 — 도시 `food` 스탯이 음수면 `stats`에 `food` 항목이 있는 건물 우선, 없으면 `production_cost`가 가장 낮은 것.
  - **연구**: 아무것도 연구 안 하고 있으면, 연구 가능(선행조건 충족, 미연구) 기술 중 `research_cost`가 가장 낮은 것.
  - **정부**: 아직 분과를 안 골랐으면 목록의 첫 번째로 즉시 선택. 한번 고른 뒤엔 절대 안 바꿈(고정).
  - **턴 진행**: 위 판단을 끝내면 매번 `nextTurnRequest`를 보내 턴을 넘김 — 다음 턴에 다시 같은 판단 반복. 무한 루프.
- **범위**: 싱글 플레이어 로컬 데모용. 멀티플레이어 여러 봇 동시 자동화는 범위 밖(각자 다른 탭에서 `?autoplay=true`로 접속하면 이론상 되긴 하지만 이번엔 신경 안 씀).

## 아키텍처

새 클라이언트 클래스 `client/src/AutoPlayController.ts` — 기존 `client/src/testing/TestUtils.ts`(접속/유닛찾기 헬퍼)와 `client/src/testing/scenarios/CitySettlement.test.ts`(정착 액션 발동 패턴)를 그대로 재사용/참고. `client/src/Index.ts`에 `autoplay=true`일 때 이 컨트롤러를 생성해 `run()` 호출하는 분기 추가.

건물/기술 카탈로그(`availableBuildings`/`availableTechnologies`/`availableGovernmentBranches`)는 컨트롤러가 시작할 때 직접 요청해서 캐싱 — 기존 UI 패널들이 이미 쓰는 것과 동일한 요청/응답 이벤트를 재사용(새 서버 코드 불필요, 이미 서브프로젝트 2a/2c/3에서 다 만들어짐).

## 범위 밖

- 새 서버 코드 없음 — 클라이언트가 기존 소켓 이벤트를 사람 대신 보내는 것뿐.
- 화면 녹화 자체(영상 파일 생성) — 사용자가 나중에 본인 화면 녹화 도구로 직접 함. 이번 작업은 "녹화할 만한 자동 진행 화면"을 만드는 것까지.
- 전투, 유닛 생산 등 아직 게임에 없는 시스템에 대한 자동화 — 있는 기능만 자동화.

## 리스크 / 열린 사항

- 정착민이 여러 개 생기는 경우(현재 시작 시 정착민 1기뿐)는 고려 안 함 — 하나만 처리.
- 무한 루프이므로 브라우저 탭을 계속 열어둬야 진행됨(백그라운드 서버 프로세스가 종료되지 않는 한 계속 감).
